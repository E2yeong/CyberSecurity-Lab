1) What Is Asymmetric Encryption?

Analogy: Think of a mailbox (public key) and the private key (the key to open it). Anyone can drop a letter into the mailbox (encrypt using public key), but only the person with the key can open it (decrypt with private key).

Advantages: Solves the key distribution problem — no need to pre-share a secret like in symmetric encryption.

  • Encryption (Confidentiality): Encrypt with recipient’s public key → only recipient can decrypt with private key.
  • Digital Signatures (Authenticity/Integrity): Sign with sender’s private key → anyone can verify with public key.

2.1 RSA

Core idea: Relies on the hardness of factoring large semiprimes.

  • Choose two large primes \(p,q\).
  • Compute \(n = pq\), \(\varphi(n) = (p-1)(q-1)\).
  • Choose \(e\), then compute \(d\) such that \(ed \equiv 1 \pmod{\varphi(n)}\).
  • Public key: \((n,e)\); Private key: \((n,d)\).
  • Encrypt: \(c = m^e \bmod n\); Decrypt: \(m = c^d \bmod n\).
Example (small numbers):
\(p=61, q=53 \Rightarrow n=3233, \varphi(n)=3120\).
\(e=17\), \(d=2753\).
Encrypt \(m=65 \Rightarrow c=65^{17}\bmod 3233=2790\).
Decrypt \(2790^{2753}\bmod 3233=65\).
(In practice, primes are hundreds of digits long.)

2.2 ECC (Elliptic Curve Cryptography)

Idea: Uses point addition/multiplication on elliptic curves over finite fields, relying on the hardness of the Elliptic Curve Discrete Logarithm Problem (ECDLP).

  • Much shorter keys provide equivalent security to RSA.
  • Efficient in performance, transmission, and storage.
  • Widely used curves: X25519 (key exchange), Ed25519 (signatures).

2.3 Diffie–Hellman / ECDH

Goal: Not direct encryption, but secure session key exchange.

  • Ephemeral keys (DHE/ECDHE) provide Perfect Forward Secrecy (PFS).
  • Used in TLS/HTTPS for negotiating symmetric session keys.

3) Hybrid Encryption in Practice

Large data is encrypted with fast symmetric ciphers (AES, ChaCha20). Asymmetric crypto is used only to exchange the symmetric session key.

Example: In TLS 1.3, ECDHE is used for key agreement, then AES-GCM or ChaCha20-Poly1305 for actual data encryption.

Modern trend: KEM (Key Encapsulation Mechanism) + KDF + AEAD. Example: HPKE standard.

4) Critical Implementation Details

  • Padding/Formats: Use RSA-OAEP for encryption, RSA-PSS for signatures. Avoid legacy PKCS#1 v1.5.
  • Side-channel resistance: Implement constant-time operations, blinding, etc.
  • ECC point validation: Ensure points are on the curve and in the correct subgroup.
  • Randomness: Keys and nonces must come from a CSPRNG.
  • Key management: Store private keys securely (HSM, TPM, KMS). Rotate/revoke as needed.
  • Disable legacy crypto: e.g., TLS 1.0, 3DES, static RSA key exchange.

5) Preparing for the Quantum Era

Problem: Shor’s algorithm breaks RSA/ECC; Grover weakens symmetric key strength.

  • Short-term: Hybrid deployment (classical + post-quantum).
  • Long-term: Transition to PQC (e.g., Kyber for encryption/KEM, Dilithium/SPHINCS+ for signatures).
  • Symmetric: AES-256 recommended.

6) Practical Checklist

  • Use TLS 1.3 with (EC)DHE + AEAD.
  • Signatures: Ed25519 or RSA-PSS.
  • Private keys in HSM/KMS only.
  • Random numbers from OS/CSPRNG (e.g., /dev/urandom, Windows CNG).
  • Test for side-channel & padding oracle attacks.
  • Disable outdated ciphers/protocols; patch regularly.
