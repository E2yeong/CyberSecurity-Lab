1) Why HTTPS?

HTTP sends data in plaintext — like mailing a postcard: anyone on the path can read or alter it. TLS (Transport Layer Security) wraps HTTP with encryption and integrity, creating HTTPS = HTTP + TLS.

Analogy: HTTP is an open postcard. HTTPS is a sealed, tamper-evident envelope with a verified return address (digital signature). Eavesdroppers can see you sent a letter (metadata), but can’t read or alter the contents.

2) TLS Core Objectives (CIA)

Confidentiality

Data is encrypted in transit so observers can’t read it.

AEAD: AES-GCM, ChaCha20-Poly1305

Integrity

Detects modification. AEAD modes authenticate ciphertext and associated data.

GCM/Poly1305 tags

Authentication

Verifies the peer’s identity via X.509 certificates and PKI signatures (CAs).

Server auth; optional client auth (mTLS)

3) TLS Handshake: 1.2 vs 1.3

High-level flow

TLS 1.2 (legacy, still seen)

  1. ClientHello: versions, cipher suites, extensions.
  2. ServerHello: choice + Certificate.
  3. Key exchange: RSA or (EC)DHE (PFS if *DHE*).
  4. Finished: switch to symmetric encryption.
Use (EC)DHE to get Perfect Forward Secrecy (PFS). Avoid static RSA key exchange.

TLS 1.3 (modern default)

  • Fewer round trips (1-RTT), faster connects.
  • Removes RSA key exchange → only (EC)DHE (PFS by default).
  • Only strong AEAD ciphers (AES-GCM, ChaCha20-Poly1305).
  • Optional 0-RTT for session resumption (replay-risk aware).
Result: simpler, faster, and more secure defaults.

Key extensions you’ll encounter

  • SNI (Server Name Indication): host selection on shared IPs.
  • ALPN: picks HTTP/1.1 vs HTTP/2 vs HTTP/3.
  • OCSP stapling: server staples revocation proof to cut latency/trust gaps.
  • ECH (Encrypted ClientHello, emerging): hides SNI from on-path observers.

4) Certificates (X.509) & the Chain of Trust

The server presents a certificate (“this public key belongs to example.com”) signed by a Certificate Authority (CA).

What’s inside a cert?

  • Subject (domain), SAN (Subject Alternative Names)
  • Public key (RSA/ECDSA), validity period
  • Issuer (CA), signature, key usage/extKeyUsage
SAN is authoritative for domain binding (CN alone is deprecated).

Trust chain

  1. Server cert → intermediate CA → root CA
  2. Browser/OS trusts the root store
  3. Validation: chain building, signature checks, hostname match, validity window

Revocation & CT

  • OCSP/CRL: revocation is imperfect (latency/reliability). OCSP stapling helps.
  • Certificate Transparency (CT) logs: browsers expect SCTs (proof of public logging).

Operational notes

  • Use automated issuance/renewal (e.g., ACME / Let’s Encrypt). Rotate keys on schedule.
  • Prefer ECDSA (P-256) or RSA-2048+ (RSA-3072 recommended for long-life).
  • Wildcard + SANs when appropriate; ensure proper key usage flags.

5) Hybrid Crypto: Asymmetric + Symmetric

TLS uses asymmetric crypto (RSA/ECDSA for signatures, (EC)DHE for key agreement) to securely establish a symmetric session key (AES-GCM/ChaCha-Poly). This blends “secure but slow” with “fast at scale.”

Perfect Forward Secrecy (PFS): Compromise of the server’s long-term key does not decrypt past sessions because each session’s ephemeral (EC)DH key was unique and discarded.

6) Modern Cipher Suites & Server Config

Recommended (TLS 1.3)

  • TLS_AES_128_GCM_SHA256
  • TLS_AES_256_GCM_SHA384
  • TLS_CHACHA20_POLY1305_SHA256 (great on mobile/CPU-constrained)

Curves: X25519, P-256

Recommended (TLS 1.2, if needed)

  • ECDHE + ECDSA/RSA with AES-GCM or CHACHA20-POLY1305
  • Disable CBC, RC4, 3DES

Enable OCSP stapling, secure renegotiation, HSTS.

Disable: SSLv2/3, TLS 1.0/1.1, export ciphers, NULL/anon suites, static RSA key exchange, RC4, 3DES, weak DH (< 2048-bit).

Example policy ideas (pseudocode)

# Pseudo-directives (web server / proxy conceptually)
tls_min_version = 1.3
tls_ciphers_13 = TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256
# If 1.2 is required:
tls_min_version = 1.2
tls_ciphers_12 = ECDHE+AESGCM:ECDHE+CHACHA20 !CBC !RC4 !3DES
curves = X25519:P-256
ocsp_stapling = on
hsts = max-age=31536000; includeSubDomains; preload
alpn = h2; http/1.1
Exact syntax differs by Nginx/Apache/Envoy/Caddy; use each vendor’s modern TLS guide.

7) Session Resumption, 0-RTT, ALPN, HTTP/2/3

  • Resumption: Tickets (1.3) or session IDs (1.2) speed reconnects; rotate ticket keys regularly.
  • 0-RTT (TLS 1.3): Enables early data on resumption but replayable. Only allow for idempotent requests and with rate-limiting.
  • ALPN: Negotiates HTTP/2 or HTTP/3 (QUIC uses TLS 1.3).
  • HTTP/2: Requires strong ciphers and ALPN; parallelism over one connection.
  • HTTP/3: Runs over QUIC (UDP) with TLS 1.3 integrated; faster handshakes, better loss recovery.

8) Browser Security: HSTS, Mixed Content, Errors

HSTS

  • HTTP Strict Transport Security forces HTTPS and guards against SSL-strip.
  • Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
  • Submit to the preload list only after verifying all subdomains support HTTPS.

Mixed Content

  • HTTPS pages must not load HTTP subresources.
  • Upgrade or block mixed content (scripts, images, iframes).

Certificate Errors

  • Expired cert, hostname mismatch, untrusted issuer → hard failures.
  • Revocation: OCSP stapling and short-lived certs mitigate gaps.
  • mTLS: client certificates for B2B, APIs, or admin portals.

9) Historic Attacks & What to Avoid

Notorious issues

  • BEAST/CRIME/BREACH: compression & CBC pitfalls → use TLS 1.3/AEAD, disable TLS-level compression.
  • POODLE: SSLv3 CBC flaw → disable SSLv3 and old TLS.
  • Heartbleed: OpenSSL memory leak → patch promptly; rotate secrets.
  • ROBOT: old RSA padding oracles → avoid legacy RSA key exchange; use TLS 1.3.
  • Logjam/DROWN/SWEET32/RC4: export/weak DH, SSLv2 bridging, 64-bit block ciphers → enforce strong params, disable legacy suites.

Operational mistakes

  • Weak DH groups (< 2048-bit) or static params reused for years.
  • Forgotten subdomains without HTTPS breaking HSTS preload.
  • Long-lived keys without rotation or poor randomness (non-CSPRNG).
  • Ignoring mixed content or certificate transparency expectations.

10) Post-Quantum Era & Migration

Shor’s algorithm breaks RSA and ECC; Grover’s algorithm roughly halves symmetric security (AES-128 ≈ 64-bit strength).

Migration path

  • Symmetric: prefer AES-256-GCM (or ChaCha20-Poly1305) for long-term confidentiality.
  • Key exchange/signatures: adopt PQC (e.g., Kyber for KEM; Dilithium/SPHINCS+ for signatures) as standards land in mainstream TLS stacks.
  • Hybrid KEX: X25519 + Kyber (or P-256 + Kyber) to hedge risk during transition.
  • Inventory your crypto, define a policy for PQC rollout, and test interop carefully.
HTTP/3 + TLS 1.3 stacks are already experimenting with hybrid key exchanges. Expect ecosystem updates in libraries, HSMs, and browsers.

11) Deployment Checklist & Testing

Server-side

  • TLS 1.3 enabled; TLS 1.2 only if required (hardened).
  • Strong cipher suites (AEAD), PFS, X25519 curve first.
  • OCSP stapling, short-lived certs, automated renewal (ACME).
  • HSTS (+ preload after verification), no mixed content.
  • Session ticket key rotation; cautious 0-RTT policy.
  • Disable SSLv2/3, TLS 1.0/1.1, export/NULL/RC4/3DES, weak DH.

Client/Browser

  • Validate hostnames; SAN-based matching.
  • Respect CT policies; show clear errors for trust failures.
  • Prefer ECH (when available) and ALPN negotiation.

Testing

  • Use comprehensive TLS testers (protocols, ciphers, HSTS, OCSP).
  • Pen-test for downgrade, replay (0-RTT), and mixed-content issues.
  • Monitor cert expiry & issuance (CT log alerts).
